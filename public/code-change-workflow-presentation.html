Hermes Skill Summary｜Hermes 技能总结

Code Change Workflow 代码修改工作流

A clear, safe, test-first process for changing software. 中文：这是一个清晰、安全、以测试优先的软件修改流程，帮助团队避免“直接开写、最后才发现问题”。

01

Core rule｜核心原则： Do not jump straight into code. 中文：不要一上来就写代码。First define behavior and success criteria, then test, implement, verify, review, commit, push, and monitor. 中文：先定义目标行为和成功标准，再测试、实现、验证、审查、提交、推送并观察结果。

When to use｜何时使用 Workflow｜流程 TDD｜测试驱动开发 Verification｜验证 Report｜汇报

When to Use ｜何时使用

Use it whenever repository files may change. 中文：只要可能改动代码仓库里的文件，就应该使用。

Use for｜适用于
  • Bug fixes｜修复错误
  • New features｜新增功能
  • UI / UX changes｜界面 / 用户体验修改
  • Refactors｜代码重构
  • Tests｜测试修改
  • Build, deployment, config changes｜构建、部署、配置修改
  • Dependency updates｜依赖更新
Do not use for｜不适用于
  • Read-only investigation｜只读调查
  • Pure explanation questions｜纯解释问题
  • One-off data analysis outside a codebase｜代码库之外的一次性数据分析

The 11-Step Workflow ｜11 步工作流

Move from understanding to deployment carefully. 中文：从理解需求到部署上线，每一步都要有证据。

Understand the request｜理解需求

Restate the bug or feature, inspect relevant files, check current behavior, and check git status. 中文：重述问题或功能，查看相关文件和当前行为，并检查 Git 状态。

Define success criteria｜定义成功标准

Write expected behavior, edge cases, tests, manual checks, and out-of-scope items before coding. 中文：写代码前先明确预期行为、边界情况、测试、人工检查和不做什么。

Create a task plan｜创建任务计划

For non-trivial work, track steps like inspect, test, implement, verify, review, commit, push, and monitor. 中文：复杂任务要维护任务清单。

Use TDD when practical｜尽量使用测试驱动开发

Write a failing test first, fix minimally, then make the test pass. 中文：先写会失败的测试，再做最小修复，让测试通过。

Implement minimally｜最小化实现

Change only what is needed; avoid unrelated refactors, unrelated files, and hidden product changes. 中文：只改必要内容，避免无关重构、无关文件和偷偷改变产品行为。

Verify in layers｜分层验证

Run targeted tests first, then broader tests, lint, typecheck, and build. 中文：先跑针对性测试，再跑完整测试、代码规范、类型检查和构建。

Manual localhost test｜本地人工测试

For UI or app flows, open the rendered app, perform the exact user actions, and check the browser console. 中文：界面或应用流程必须打开真实页面测试，并检查浏览器控制台。

Independent review｜独立审查

Review git diff, confirm scope, check for secrets, and use a second review pass for meaningful changes. 中文：审查差异、确认范围、检查秘密信息，重要改动要做第二轮审查。

Commit cleanly｜干净提交

Stage only intended files and use clear conventional commit messages. 中文：只暂存目标文件，并使用清晰的规范化提交信息。

Push and create PR｜推送并创建合并请求

Push only after validation; include summary, tests, manual notes, screenshots, and caveats. 中文：验证后再推送；PR 中写明总结、测试、人工验证、截图和限制。

Monitor CI and deployment｜观察 CI 和部署

Check failed logs, fix issues, re-push, and verify deployment aliases when needed. 中文：检查失败日志、修复、重新推送，并在需要时验证部署域名。

TDD: Red → Green → Refactor ｜测试驱动开发：红 → 绿 → 重构

The safest path for bugs and behavior changes. 中文：修 bug 和改行为时最安全的方法。

Red｜红： write failing test｜写失败测试 → Green｜绿： smallest fix｜最小修复 → Refactor｜重构： clean after passing｜通过后清理
1

Prove the bug｜证明问题存在

A failing test shows the current bug is real. 中文：失败测试证明当前问题确实存在。

2

Fix only enough｜只修必要部分

Make the smallest safe change that passes the test. 中文：做最小且安全的修改，让测试通过。

3

Prevent regression｜防止回归

The test protects the behavior in the future. 中文：测试会保护未来不再犯同样错误。

Verification Checklist ｜验证清单

Do not call the work done until these are checked or clearly marked blocked. 中文：除非完成或明确说明阻塞，否则不要说任务完成。

  • Request understood｜已理解需求
  • Git branch and status checked｜已检查 Git 分支和状态
  • Success criteria defined｜已定义成功标准
  • Failing test written or exception explained｜已写失败测试，或解释为何不适用
  • Minimal implementation completed｜已完成最小实现
  • Targeted tests pass｜针对性测试通过
  • Full relevant tests pass or failures documented｜相关完整测试通过，或记录失败原因
  • Lint, typecheck, and build pass or caveats documented｜代码规范、类型检查、构建通过，或记录限制
  • Manual UI / app test completed if relevant｜如涉及界面，已人工测试
  • Browser console checked｜已检查浏览器控制台
  • Diff reviewed for scope and secrets｜已审查差异范围和秘密信息
  • Independent review completed for meaningful changes｜重要改动已独立审查
  • Intended files committed｜目标文件已提交
  • Branch pushed / PR created if requested｜如需要，已推送分支或创建 PR
  • CI and deployment checked when applicable｜如适用，已检查 CI 和部署

Common Pitfalls ｜常见坑

These are the mistakes the workflow is designed to prevent. 中文：这个工作流就是为了避免这些错误。

Pitfall｜坑

Coding too early｜过早写代码

Starting before defining behavior usually causes rework. 中文：没定义目标行为就开始写，通常会返工。

Pitfall｜坑

No regression test｜没有回归测试

A fix without a test can break again silently. 中文：没有测试的修复，未来可能悄悄坏掉。

Pitfall｜坑

Only happy-path checks｜只测顺利路径

Edge cases and failure states matter. 中文：边界情况和失败状态同样重要。

Pitfall｜坑

Unrelated files｜提交无关文件

Always inspect the diff and stage intentionally. 中文：一定检查差异，只提交目标文件。

Pitfall｜坑

Localhost ≠ live｜本地不等于线上

If the user asks for live, verify production or preview deployment. 中文：如果用户问线上效果，要验证生产或预览部署。

Pitfall｜坑

No CI follow-up｜不跟进 CI

After pushing, check automated checks and deployment. 中文：推送后要检查自动化测试和部署。

Final Report Format ｜最终汇报格式

What a good completion update should include. 中文：一个好的完成汇报应该包含这些内容。

“Done” means verified, reviewed, and safe to ship. 中文：“完成”意味着已经验证、审查，并且可以安全交付。
Include｜包含
  • Bug / feature description｜问题 / 功能描述
  • Files changed｜修改的文件
  • What was implemented｜实现了什么
  • Tests and results｜测试和结果
  • Build / lint / typecheck results｜构建 / 规范 / 类型检查结果
  • Manual test notes｜人工测试记录
  • Review findings｜审查发现
  • Commit / PR / push status｜提交 / PR / 推送状态
  • Remaining caveats｜剩余限制或风险
